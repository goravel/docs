## Yangilash Qo‘llanmasi

[Golang v1.24 endi qo‘llab-quvvatlanmaydi](https://endoflife.date/go) sababli, Goravel ning qo‘llab-quvvatlaydigan Golang versiyasi 1.24 dan 1.25 ga yangilandi.

goravel/example loyihasini v1.17 dan v1.18 ga o‘tkazish PR yangilash uchun namuna sifatida foydalanish mumkin: [goravel/example#133](https://github.com/goravel/example/pull/133).

Siz quyidagini AI kod yozish agentingizga nusxalab, Goravel loyihangizni avtomatik ravishda yangilashingiz mumkin.

````markdown
# Upgrade Guide for Goravel v1.18

**Before you begin**: ensure all changes are committed or stashed, the project builds cleanly (`go build ./...`), and Go ≥ 1.25 is installed, Goravel version ≥ 1.17.0.

## Step 1: Update Go module dependencies

**Detection**: Check which facade packages are already installed:

```shell
rg -l 'goravel/(gin|fiber|redis|s3|oss|cos|minio)' go.sum 2>/dev/null
```

**Action**: Run the framework upgrade command and upgrade only the facade packages detected in `go.sum`:

```shell
go get github.com/goravel/framework@latest

# For each facade found in detection, run the corresponding go get.
# If goravel/gin is in go.sum:   go get github.com/goravel/gin@latest
# If goravel/fiber is in go.sum: go get github.com/goravel/fiber@latest
# If goravel/redis is in go.sum: go get github.com/goravel/redis@latest
# If goravel/s3 is in go.sum:    go get github.com/goravel/s3@latest
# If goravel/oss is in go.sum:   go get github.com/goravel/oss@latest
# If goravel/cos is in go.sum:   go get github.com/goravel/cos@latest
# If goravel/minio is in go.sum: go get github.com/goravel/minio@latest

go mod tidy
```

## Step 2: Migrate HTTP middleware from func to interface

`contracts/http.Middleware` changed from `func(http.Context)` to an interface with `Handle(http.Context)` and `Signature() string` methods. Every custom middleware must be updated.

### 2a. Detect all custom middleware functions

```shell
rg -n 'func\s+\w+\s*\(.*\)\s+http\.Middleware' --type go
rg -n 'func\s+\w+\s*\(.*\)\s+contractshttp\.Middleware' --type go
```

Also search for anonymous middleware (inline `return func(ctx http.Context)`):

```shell
rg -n 'return\s+func\(ctx\s+http\.Context\)' --type go
```

### 2b. Convert each middleware

For each detected middleware, keep the factory function signature and replace the closure body with a struct literal:

```diff
 func Auth() http.Middleware {
-    return func(ctx http.Context) {
-        ctx.Request().Next()
-    }
+    return &Auth{}
 }

+type Auth struct{}
+
+func (a *Auth) Handle(ctx http.Context) {
+    ctx.Request().Next()
+}
+
+func (a *Auth) Signature() string {
+    return "auth"
+}
```

If the original factory accepted parameters, pass them to the struct literal:

```diff
 func Throttle(name string) http.Middleware {
-    return func(ctx http.Context) {
-        // throttle logic using name
-    }
+    return &Throttle{Name: name}
 }
+
+type Throttle struct{ Name string }
+
+func (t *Throttle) Handle(ctx http.Context) { /* throttle logic using t.Name */ }
+func (t *Throttle) Signature() string { return "throttle-" + t.Name }
```

Call sites like `middleware.Auth()` stay unchanged — the factory function still returns `http.Middleware`, but now returns a struct instead of a closure.

### 2c. Verification

```shell
# Ensure no remaining func-returning middleware patterns
rg -n 'func.*http\.Middleware' --type go

# Build must succeed
go build ./...
```

## Step 3: Update validation rule and filter maps

Validation rules and filters now use `map[string]any` instead of `map[string]string`.

### 3a. Detect affected files

```shell
rg -n 'map\[string\]string.*Validate\(' --type go
rg -n 'map\[string\]string.*Make\(' --type go
rg -n 'Rules\(\).*map\[string\]string' --type go
rg -n 'Filters\(\).*map\[string\]string' --type go
rg -n 'Messages\(\).*map\[string\]string' --type go
rg -n 'Attributes\(\).*map\[string\]string' --type go
```

Also search for `Validate` and `Make` calls that may already use `map[string]any` or use short variable declarations:

```shell
rg -n '\.Validate\(.*map\[string\]' --type go
rg -n 'validation\.Make\(.*map\[string\]' --type go
```

### 3b. Apply the transformation

Replace `map[string]string` with `map[string]any` in `Validate` and `Make` calls:

```diff
-ctx.Request().Validate(map[string]string{
+ctx.Request().Validate(map[string]any{
     "title": "required|max:255",
 })
```

Also update form request method signatures — `Rules()`, `Messages()`, `Attributes()`, and `Filters()`:

```diff
-func (r *UserCreate) Rules(ctx http.Context) map[string]string {
-    return map[string]string{
+func (r *UserCreate) Rules(ctx http.Context) map[string]any {
+    return map[string]any{
         "name": "required",
     }
 }

-func (r *UserCreate) Messages(ctx http.Context) map[string]string {
-    return map[string]string{}
+func (r *UserCreate) Messages(ctx http.Context) map[string]any {
+    return map[string]any{}
 }

-func (r *UserCreate) Filters(ctx http.Context) map[string]string {
-    return map[string]string{
+func (r *UserCreate) Filters(ctx http.Context) map[string]any {
+    return map[string]any{
         "name": "trim",
     }
 }
```

If a `regex` or `not_regex` rule value contains `|` and is followed by more rules, switch to `[]string`:

```diff
 rules := map[string]string{
-    "slug": "required|regex:^(news|docs)-[a-z]+$|string",
+    "slug": []string{"required", "regex:^(news|docs)-[a-z]+$", "string"},
 }
```

### 3c. Verification

```shell
# Ensure no remaining map[string]string in validation contexts
rg -n 'Validate\(.*map\[string\]string' --type go
rg -n 'Make\(.*map\[string\]string' --type go
rg -n 'Rules\(.*map\[string\]string' --type go
rg -n 'Messages\(.*map\[string\]string' --type go
rg -n 'Filters\(.*map\[string\]string' --type go

go build ./...
```

## Step 4: Update mail template content fields

Mail content fields are now explicit: `HtmlView` for HTML templates, `TextView` for plain-text templates, `Html`/`Text` for raw body content.

### 4a. Detect affected files

```shell
rg -n '\.View\s*:' --type go -g '!vendor/**' -g '!storage/**'
rg -n 'Content\(.*mail\.Content\{' --type go
rg -n '&mail\.Content\{' --type go
```

### 4b. Apply the transformation

For each `mail.Content{}` literal:

| Old field | New field | Condition |
|-----------|-----------|-----------|
| `View:` | `HtmlView:` | Always (was for HTML templates) |
| `Text:` (value ends in `.html`, `.txt`, `.tmpl`, or is a file path) | `TextView:` | When the value was a template path |
| `Text:` (value is a raw string, not a path) | `Text:` | Keep as-is for raw body |

**Example — template-based email**:
```diff
 Content(mail.Content{
-    View: "welcome.html",
-    Text: "welcome.txt",
+    HtmlView: "welcome.html",
+    TextView: "welcome.txt",
     With: map[string]any{
         "Name": "Goravel",
     },
 })
```

**Example — raw body email (no change)**:
```go
Content(mail.Content{Text: "Hello Goravel"})
```

If a file uses `Html:` / `Text:` already (raw body pattern from v1.17+), no change is needed.

### 4c. Verification

```shell
# Ensure no remaining View: in mail Content structs
rg -n '\.View\s*:.*\.html' --type go

go build ./...
```

## Step 5: Rename gRPC config key servers to clients

### 5a. Detect affected files

```shell
rg -n '"servers"' --type go -g '*grpc*'
rg -n 'grpc\.servers' --type go
```

### 5b. Apply the transformation

In `config/grpc.go`, rename the config key:

```diff
 config.Add("grpc", map[string]any{
     "host": config.Env("GRPC_HOST", ""),
-    "servers": map[string]any{
+    "clients": map[string]any{
         "user": map[string]any{
             "host":           config.Env("GRPC_USER_HOST", ""),
             ...
         },
     },
 })
```

In application code, update any config path references:

```diff
-facades.Config().Get("grpc.servers.user.host")
+facades.Config().Get("grpc.clients.user.host")
```

### 5c. Verification

```shell
# Ensure no remaining "servers" key in grpc config
rg -n '"servers"' --type go -g '*grpc*'
rg -n 'grpc\.servers' --type go

go build ./...
```

## Step 6: Update validation error messages and custom rule names

The new native validation engine ([goravel/framework#1397](https://github.com/goravel/framework/pull/1397)) produces Laravel-style error messages and evaluates all rules (including wildcards) differently. Any tests that assert on specific validation error messages or JSON error responses must be updated.

### 6a. Detect affected tests

```shell
rg -n 'Equal.*message.*required' --type go
rg -n 'Equal.*message.*regex' --type go
rg -n 'Equal.*message.*date' --type go
rg -n 'Required.*is required to not be empty' --type go
```

### 6b. Update error message expectations

Replace old-style error message assertions with the new Laravel-style messages:

| Old message | New message |
|-------------|-------------|
| `"name is required to not be empty"` | `"The name field is required."` |
| `"code value does not pass the regex check"` | `"The code field format is invalid."` |
| `"date value should be a date string"` | `"The date field must be a valid date."` |

Example test diff:

```diff
-s.Equal("{\"message\":{\"name\":{\"required\":\"name is required to not be empty\"}}}", content)
+s.Equal("{\"message\":{\"name\":{\"required\":\"The name field is required.\"}}}", content)
```

> **Note**: Wildcard rules (e.g., `scores.*`, `tags.*`) now consistently appear in error output even when no input is provided. Add assertions for these if your tests check the full error response body.

### 6c. Rename custom rules that conflict with built-in rules

The new engine includes built-in rules (`required`, `email`, `exists`, `unique`, etc.) that may conflict with custom rule names. Rename any custom rule whose `Signature()` matches a built-in rule name:

```diff
 func (receiver *Exists) Signature() string {
-    return "exists"
+    return "exists_custom"
 }
```

### 6d. Verification

```shell
go build ./...
go test ./... 2>&1 || true
```

## Step 7: Install the goravel-development skill

Download the Goravel AI coding conventions skill from goravel-lite so you can reference project structure, facades, testing patterns, and Artisan commands during the upgrade:

```shell
mkdir -p .agents/skills/goravel-development
curl -fsSL -o .agents/skills/goravel-development/SKILL.md \
  https://raw.githubusercontent.com/goravel/goravel-lite/master/.agents/skills/goravel-development/SKILL.md
```

### 7a. Verification

```shell
ls -l .agents/skills/goravel-development/SKILL.md
```

## Step 8: Final verification

After completing all steps, run the full verification suite:

```shell
go build ./...
go vet ./...
go test ./... 2>&1 || true
```

If tests fail, the most common causes are:
- Test middleware functions that were not migrated (step 2).
- Test validation calls using `map[string]string` (step 3).
- Test mail Content structs with old field names (step 4).
- Test validation error message assertions with old format (step 6).
- Custom rule signatures conflicting with built-in rule names (step 6).
````

### AI SDK

Goravel v1.18 suhbatlar, biriktirilgan fayllar, yaratilgan media va nutq ish oqimlari uchun birinchi tomon AI SDK ni bitta fasad orqali taqdim etadi. Fasadni `go run .artisan package:install AI` bilan o'rnating, so'ngra `goravel/openai`, `goravel/anthropic` yoki `goravel/gemini` kabi provayder paketini o'rnating.

Birinchi versiya quyidagilarni o'z ichiga oladi:

- Agent suhbatlari `make:agent`, `make:tool`, sozlanishi mumkin bo'lgan AI generator yo'llari, `Prompt`, `Stream`, vosita chaqiruvi, prompt middleware, provayder failover.
- So'rov doirasidagi hujjat va tasvir biriktirilmalari baytlar, o'quvchilar, yo'llar, saqlash, URL'lar, yuklamalar yoki provayder boshqaruvidagi fayl ID'laridan.
- Tasvir va audio generatsiyasi `Store` / `StoreAs` yordamchilari bilan, shuningdek, til va diarizatsiya opsiyalari bilan transkripsiya.

`goravel-lite` skafoldi, shuningdek, AI kodlash agentlarini (Cursor, Copilot, Codex va boshqalar) o'rgatadigan _AI Agent Development Skill_ ni o'z ichiga oladi. Goravel loyihalari bilan qanday ishlash kerak. Ko‘nikma loyiha tuzilishi, fasadlar, marshrutlash, ORM, test qilish konvensiyalari va Artisan skafoldlash buyruqlarini o‘z ichiga oladi. AI agent ushbu ko‘nikmani o‘qiganda, u Goravel loyihasini ishga tushirishni, fasadlarni qo‘shish uchun `package:install` dan foydalanishni, fasad namunasi bilan ishlashni, `testify` to‘plamlari va mock yordamchilari bilan test yozishni va Artisan `make:*` generatorlaridan foydalanishni biladi. Siz `.agents/skills/goravel-development/CUSTOM.md` faylini yaratish orqali ko‘nikmani loyihaga xos qoidalar bilan kengaytirishingiz mumkin. Ko‘nikma `goravel-lite` bilan birga versiyalanadi va freymvorkni yangilaganingizda yangilanadi.

Minimal so‘rov shunday ko‘rinadi:

```go
conversation, err := facades.AI().Agent(&agents.SupportAgent{})
if err != nil {
    return err
}

response, err := conversation.Prompt("How do I create a controller?")
if err != nil {
    return err
}

fmt.Println(response.Text())
```

### Telemetriya

Goravel v1.18 OpenTelemetry asosida qurilgan telemetriya modulini taqdim etadi, u izlar, ko‘rsatkichlar va jurnallarni yig‘ib, ularni Jaeger, Prometheus, Grafana yoki Datadog kabi har qanday OTLP-mos keluvchi backendga eksport qiladi. Fasadni `go run . bilan o‘rnating. artisan to‘plami: Telemetry` ni o‘rnating.

Dastlabki versiya quyidagilarni o‘z ichiga oladi:

- Izlar, ko‘rsatkichlar va jurnallar bitta `config/telemetry.go` faylidan sozlangan bo‘lib, OTLP, konsol va maxsus eksport qiluvchilar, namuna olish va partiyalash nazoratlari bilan.
- HTTP server, HTTP mijoz, gRPC, ma'lumotlar bazasi va logger uchun o‘rnatilgan instrumentatsiya, xizmatlar bo‘ylab iz kontekstining tarqalishi bilan.
- Ilova to‘xtaganda avtomatik flush va o‘chirish.

Qo‘lda yaratilgan span quyidagicha ko‘rinadi:

```go
ctx, span := facades.Telemetry().Tracer("app").Start(ctx, "process-order")
defer span.End()

span.SetAttributes(telemetry.String("order.id", "1234"))

// Pass ctx down to create child spans
processItems(ctx)
```

`Tracer` argument (`"app"`) bu kodni aniqlovchi asbobsozlik doirasi nomi bo'lib, u `otel.scope.name` sifatida qayd etiladi. An'anaga ko'ra, bu paket import yo'li yoki amaliy kod uchun ilova nomidir.

### Laravel uslubidagi kolleksiyalar

Goravel endi `support/collect` paketida ravon (fluent) ishtiyoqli va kechiktirilgan kolleksiya yordamchilarini taqdim etadi. Siz bo'laklardan kolleksiyalar yaratishingiz, filtrlash va xaritalash operatsiyalarini zanjirlashingiz, strukturalar uchun Laravel uslubidagi `Where` taqqoslashlarini ishlatishingiz, qiymatlarni jamlashingiz va katta ma'lumotlar to'plamlarini `LazyCollection` bilan qayta ishlashingiz mumkin.

```go
numbers := collect.New(1, 2, 3, 4, 5, 6)

evens := numbers.Filter(func(n int, _ int) bool {
    return n%2 == 0
})

evens.All() // []int{2, 4, 6}
```

### Mahalliy tekshirish mexanizmi va o'rnatilgan qoidalar

Goravel endi avvalgi tashqi validatsiya bog'liqligi o'rniga mahalliy validatsiya dvigatelidan foydalanadi. Yangi dvigatel nuqta yozuvini, wildcard validatsiyasini, tasdiqlangan ma'lumotlarni olish, maxsus filtrlar, aniq xabar ustuvorligi va kengroq Laravel uslubidagi o'rnatilgan qoidalar to'plamini qo'llab-quvvatlaydi.

```go
validator, err := facades.Validation().Make(ctx,
    map[string]any{
        "email": " User@Goravel.dev ",
        "scores": []int{1, 2},
    },
    map[string]any{
        "email": "required|email",
        "scores.*": "required|integer",
    },
    validation.Filters(map[string]any{
        "email": "trim|lower",
    }),
)

validated := validator.Validated()
scores := validated["scores"].([]int)
```

### Validatsiya tarjimaga asoslangan xato xabarlari va lang:publish buyrug'ini qo'llab-quvvatlaydi.

Validatsiya xato xabarlari endi frameworkning [mahalliylashtirish](../digging-deeper/localization.md) tizimi tomonidan boshqariladi. Standart xabarlar `lang/en/validation.json` sifatida kiritilgan va `validation.*` tarjima kalitlaridan foydalanadi (masalan, `validation.required`, `validation.min.string`).

Yangı `lang:publish` Artisan buyrug'idan foydalanib, validatsiya til fayllarini ilovangizga nashr qiling:

```shell
go run . artisan lang:publish
go run . artisan lang:publish --force
```

Nashr qilingandan so'ng, standart xato xabarlarini sozlash uchun `lang/en/validation.json` ni tahrirlang. Boshqa tillar uchun qo'shimcha til kataloglarini yarating (masalan, `lang/zh/validation.json`). Maxsus qoida xabarlari endi `:value` va `:optionN` o'rnini bosuvchi joylarni qo'llab-quvvatlaydi.

[Hujjatni ko'rish](../the-basics/validation.md#translation--localization)

### Xizmat ko'rsatish rejimi buyruqlari

Goravel endi ilova xizmat ko'rsatish rejimi uchun `artisan down` va `artisan up` buyruqlarini taqdim etadi. Odatiy bo'lib `file` xizmat ko'rsatish haydovchisi ishlatiladi, shuning uchun `down` buyrug'i xizmat ko'rsatish metama'lumotlarini saqlashga yozadi va `up` buyrug'i uni olib tashlaydi.

```shell
go run . artisan down
go run . artisan up
```

`down` buyrug'i `--reason`, `--status`, `--redirect`, `--render`, `--secret` va `--with-secret` kabi parametrlar bilan javobni moslashtirishni qo'llab-quvvatlaydi.

```shell
go run . artisan down --reason="Upgrading database" --status=503
go run . artisan down --redirect=/maintenance
go run . artisan down --render=errors/503
go run . artisan down --with-secret
```

Xizmat ko'rsatish sozlamalari `config/app.go` faylidagi `app.maintenance` dan o'qiladi. Yangi Route o'rnatmalari ushbu konfiguratsiyani avtomatik ravishda qo'shadi. Mavjud ilovalar uni qo'lda qo'shishi mumkin:

```go
"maintenance": map[string]any{
    "driver": config.Env("APP_MAINTENANCE_DRIVER", "file"),
    "store":  config.Env("APP_MAINTENANCE_STORE", ""),
},
```

Ko'p serverli joylashtirishlar uchun, xizmat ko'rsatish holatini umumiy kesh omborida saqlash uchun `APP_MAINTENANCE_DRIVER=cache` ni o'rnating. Nomlangan kesh omborini, masalan Redis, tanlash uchun `APP_MAINTENANCE_STORE` dan foydalaning:

```ini
APP_MAINTENANCE_DRIVER=cache
APP_MAINTENANCE_STORE=redis
```

### Runners konfiguratsiya opsiyasi orqali o'chirilishi mumkin.

Goravel v1.18 ichki `app.auto_run` bayrog'ini ommaviy, glob-nazorat qiluvchi `app.disabled_runners` konfiguratsiya opsiyasi bilan almashtiradi. Bu sizga asosiy jarayonda avtomatik ishga tushiriladigan [ijrochilar](../architecture-concepts/service-providers.md#ijrochilar)ni tanlab o'tkazib yuborish imkonini beradi, boshqa xizmatlarni ishlab turgan holda — masalan, HTTP serverni o'chirmasdan, maxsus konteynerda rejalashtiruvchini ishga tushiring yoki `goravel:http`, `goravel:queue` va `goravel:grpc` ijrochilarini o'tkazib yuborish orqali faqat rejalashtiruvchi konteynerni yarating.

Qiymat Go tilidagi [`path.Match`](https://pkg.go.dev/path#Match) yordamida har bir ijrochining imzosiga mos keladigan glob naqshlarining kesimidir.

```go
// config/app.go
"app": map[string]any{
    "env":   "production",
    "debug": false,
    "disabled_runners": []string{"goravel:schedule"},
},
```

Framework ijrochi imzolari `goravel:http`, `goravel:grpc`, `goravel:queue`, `goravel:schedule` va `goravel:telemetry`dir. Naqshlar tartib bo'yicha baholanadi va birinchi mos kelgan g'alaba qozonadi, shuning uchun siz aniq va jokerli mosliklarni aralashtirishingiz mumkin.

```go
// Web container — only skip the scheduler
"disabled_runners": []string{"goravel:schedule"},

// Scheduler-only container — disable http, queue, grpc
"disabled_runners": []string{"goravel:http", "goravel:queue", "goravel:grpc"},

// Disable every framework runner
"disabled_runners": []string{"goravel:*"},

// Kill switch: run no auto-run runners in the main process
"disabled_runners": []string{"*"},
```

Notoʻgʻri naqshlar ogohlantirish sifatida qayd etiladi va runner ishlashda qoldiriladi, shuning uchun konfiguratsiyadagi xatolik hech qachon bootʻni buzmaydi. Filtrlash markaziy ravishda `foundation.configureRunners()` da ishlaydi, shuning uchun kelajakda qoʻshilgan har qanday yangi runner bir xil xatti-harakatni bepul oladi.

[Hujjatni koʻring](../getting-started/configuration.md#disabled-runners)

### Marshrutlar uchun WithoutMiddleware

Goravel v1.18 marshrutlar va marshrut guruhlari uchun `WithoutMiddleware` metodini taqdim etadi, xuddi Laravelʻning `withoutMiddleware()` kabi. Bu maʻlum marshrutlarga ota-guruhlar tomonidan qoʻllaniladigan baʻzi middlewareʻlarni chetlab oʻtish imkonini beradi — webhook nuqtalari, umumiy APIʻlar yoki autentifikatsiya yoki tezlik cheklashni oʻtkazib yuborishi kerak boʻlgan marshrutlar uchun foydali.

Middleware guruhidagi alohida marshrutlarda middlewareʻni chiqarib tashlash:

```go
facades.Route().Middleware(middleware.Auth(), middleware.Throttle("global")).Group(func(router route.Router) {
  router.Get("/dashboard", dashboardController.Index)

  // This route excludes the throttle middleware
  router.Get("/api/webhook", webhookController.Handle).
    WithoutMiddleware(middleware.Throttle("global"))
})
```

Guruhdagi barcha marshrutlar uchun middleware-ni istisno qiling:

```go
facades.Route().Middleware(middleware.Auth()).
  WithoutMiddleware(middleware.Auth()).
  Group(func(router route.Router) {
    router.Get("/public", publicController.Index)
  })
```

> **Eslatma**: Middleware istisnosi middleware-larni aniqlash uchun `Signature()` metodidan foydalanadi. `WithoutMiddleware` to'g'ri ishlashi uchun har bir middleware noyob imzo qaytarishiga ishonch hosil qiling. O'rnatilgan framework middleware-lari allaqachon noyob imzolarni taqdim etadi.

### Artisan konsoli Shutdownable orqali muvaffaqiyatli o'chirishni qo'llab-quvvatlaydi.

Konsol buyruqlari endi ixtiyoriy `console.Shutdownable` interfeysini amalga oshirish orqali xotirjam o‘chirishni tanlashi mumkin. Framework har bir buyruq orqali `signal.NotifyContext` ni ulaydi, shuning uchun `Ctrl+C` (yoki `SIGTERM` yuborish) `Handle` ga uzatilgan kontekstni bekor qiladi. Agar buyruq `Shutdown(ctx Context) error` ni ham amalga oshirsa, framework uni yangi kontekst bilan chaqiradi va jarayon chiqishidan oldin tozalash uchun 30 soniyalik byudjet beradi. `Handle` goroutine da ishlaydi, shuning uchun signal javobi darhol bo‘ladi.

`console.Context` endi `context.Context` ni o‘z ichiga oladi, shuning uchun buyruqlar `<-ctx.Done()` dan to‘g‘ridan-to‘g‘ri foydalanishi, `ctx` ni `context.Context` kutadigan funksiyalarga uzatishi va `ctx.Deadline()` / `ctx.Err()` / `ctx.Value(key)` ni kirish moslamasisiz chaqirishi mumkin. `schedule:run` buyrug‘i kanonik misoldir — u tozalashni `schedule.Shutdown(ctx)` ga yo‘naltirish uchun `Shutdown` ni amalga oshiradi.

```go
package commands

import (
  "errors"
  "net/http"

  "github.com/goravel/framework/contracts/console"
  "github.com/goravel/framework/contracts/console/command"
)

type Serve struct {
  server *http.Server
}

func (r *Serve) Signature() string   { return "serve" }
func (r *Serve) Description() string { return "Start the HTTP server" }
func (r *Serve) Extend() command.Extend {
  return command.Extend{Category: "server"}
}

func (r *Serve) Handle(ctx console.Context) error {
  if err := r.server.ListenAndServe(); err != nil && !errors.Is(err, http.ErrServerClosed) {
    return err
  }
  return nil
}

func (r *Serve) Shutdown(ctx console.Context) error {
  ctx.Info("received signal, shutting down gracefully...")
  return r.server.Shutdown(ctx)
}
```

`Shutdownable`ni amalga oshirmaydigan buyruqlar asl harakatni saqlab qoladi — signal qabul qilinganda jarayon darhol tugaydi.

[Hujjatni ko'ring](../digging-deeper/artisan-console.md#graceful-shutdown)

### Navbat drayverlari DriverWithReceive orqali guruhli iste'molni qo'llab-quvvatlaydi

Maxsus navbat drayverlari endi bloklovchi guruhli xabar iste'molini qo'llab-quvvatlash uchun `DriverWithReceive` interfeysini (ixtiyoriy ravishda) amalga oshirishi mumkin ([goravel/framework#1435](https://github.com/goravel/framework/pull/1435)) Drayver ushbu interfeysni amalga oshirganda, navbat ishchisi avtomatik ravishda `Pop` o'rniga `Receive` dan foydalanadi, bu Kafka yoki RabbitMQ kabi yuqori o'tkazuvchanlikdagi drayverlar uchun so'rov yuborish yukini kamaytiradi.

```go
import (
  "context"
  "github.com/goravel/framework/contracts/queue"
)

type KafkaDriver struct{}

// Asosiy Driver interfeysi metodlari (majburiy)
func (d *KafkaDriver) Driver() string                        { return "kafka" }
func (d *KafkaDriver) Push(task queue.Task, queue string) error { return nil }
func (d *KafkaDriver) Pop(queue string) (queue.ReservedJob, error) { return nil, nil }

// Guruhli iste'mol uchun ixtiyoriy DriverWithReceive
func (d *KafkaDriver) Receive(ctx context.Context, queue string, count int) ([]queue.ReservedJob, error) {
  // bloklovchi guruhli qabul qilish `count` tagacha ishlarni, ctx tugaganda qaytaradi
}
```

`Receive` mavjud bo'lganda, ishchi 5 soniyali har bir chaqiruv uchun taym-aut va xatoliklar yoki bo'sh guruhlar uchun eksponensial orqaga chekinish (100ms–3.2s) bilan bloklovchi guruhli siklini ishlatadi. `context.Context` ishchi to'xtatilganda bekor qilinadi, bu toza tugatishni ta'minlaydi. Mavjud drayverlar uchun hech qanday dastur kodini o'zgartirish talab qilinmaydi.

### Fiber drayver konteksti Fiber misoli hayotiy tsiklidan ajratildi

Fiber drayveri bilan so'rov muddati tugaganda, `context.Context` interfeys usullari (`Done()`, `Deadline()`, `Err()`) asosiy `fiber.Ctx` ga topshirildi, bu so'rov fasthttp tomonidan qayta ishlanganidan keyin yaroqsiz bo'ldi. Bu fon goroutinlari (masalan, `database/sql`) so'rov kontekstiga havolani saqlab qolganda nil-pointer panikalariga sabab bo'ldi ([goravel/goravel#951](https://github.com/goravel/goravel/issues/951)).

Ilova kodida hech qanday o'zgartirish talab qilinmaydi.

### Fiber drayveri shablon konfiguratsiyasi DefaultTemplate bilan almashtirildi

`LoadViewsFrom` orqali paket ko'rinishlarini qo'llab-quvvatlash uchun Fiber drayveri ([goravel/fiber#273](https://github.com/goravel/fiber/pull/273)) `gofiber/template/html/v3` ga bog'liqligini o'rniga o'rnatilgan `DefaultTemplate()` bilan almashtirdi, u `resources/views` va barcha ro'yxatdan o'tgan paket ko'rinishlarini avtomatik ravishda yuklaydi. Agar sizning `config/http.go` faylingizda `html.New(path.Resource("views"), ".tmpl")` yordamida maxsus `template` konfiguratsiyasi mavjud bo'lsa va siz [Paket ko'rinishlari LoadViewsFrom orqali](#package-views-via-loadviewsfrom) xususiyatidan foydalanmoqchi bo'lsangiz, uni `goravelfiber.DefaultTemplate()` bilan almashtiring:

```diff
 // config/http.go — fiber drayveri

import goravelfiber "github.com/goravel/fiber"

 "template": func() (fiber.Views, error) {
-    return html.New(path.Resource("views"), ".tmpl"), nil
+    return goravelfiber.DefaultTemplate(), nil
 },
```

Maxsus ajratgichlar yoki FuncMap uchun `goravelfiber.NewTemplate(goravelfiber.RenderOptions{...})` dan foydalaning:

```go
import goravelfiber "github.com/goravel/fiber"

"template": func() (fiber.Views, error) {
    return goravelfiber.NewTemplate(goravelfiber.RenderOptions{
        Delims: &goravelfiber.Delims{Left: "{[", Right: "]}"},
    })
},
```

Agar `template` konfiguratsiyasi aniqlanmagan bo'lsa, `DefaultTemplate()` avtomatik ravishda ishlatiladi — hech qanday kod o'zgarishi talab qilinmaydi.

### Validatsiya mexanizmi nativ ravishda qayta yozildi

Validatsiya mexanizmi [goravel/framework#1397](https://github.com/goravel/framework/pull/1397) da qayta yozildi — tashqi `gookit/validate` bog'liqligini Laravel validatsiya API bilan to'liq mos keladigan mahalliy amalga oshirish bilan almashtirdi. Bu o'zgarish bir qancha quyi oqim effektlarini keltirib chiqaradi:

1. **Xato xabari formati o‘zgartirildi** eski uslubdagi xabarlardan (masalan, `"name maydoni bo‘sh bo‘lmasligi kerak"`) Laravel uslubidagi xabarlarga (masalan, `"The name field is required."`). Muayyan tasdiqlash xato xabarlariga asoslanadigan testlar yangilanishi kerak.

2. **Yulduzcha qoidalari (`*.`) endi doimiy ravishda xatolarni bildiradi.** Eski dvigatel kiritishga qarab yulduzcha maydon xatolarini o‘tkazib yuborishi mumkin edi; yangi dvigatel ularni doimo baholaydi va bildiradi.

3. **Forma so‘rov metodining imzolari o‘zgartirildi** — `Rules()`, `Messages()`, `Attributes()` va `Filters()` endi `map[string]any` qaytaradi, `map[string]string` o‘rniga.

4. **Maxsus tasdiqlash qoida nomlari** yangi o‘rnatilgan qoidalar bilan ziddiyatga kelishi mumkin. O‘rnatilgan qoidalar bilan nomi bir xil bo‘lgan maxsus qoidalarni qayta nomlang (masalan, `exists` → `exists_custom`).

Xato xabarlari endi [mahalliylashtirish](../digging-deeper/localization.md) tizimi tomonidan taqdim etiladi. Standart xabarlar `lang/en/validation.json` sifatida kiritilgan va `validation.*` tarjima kalitlaridan foydalanadi. `go run . dan foydalaning artisan lang:publish` nashr qilish va moslashtirish uchun.

### gRPC konfiguratsiya kaliti servers dan clients ga o'zgartirildi

```diff
 })
```

Ilova kodida, konfiguratsiya yo'li havolalarini `grpc.servers` dan `grpc.clients` ga yangilang:

```diff
-facades.Config().Get("grpc.servers.user.host")
+facades.Config().Get("grpc.clients.user.host")
```

### HTTP drayveri timeut middleware'i bir vaqtning o'zida xavfsizlik uchun qayta yozildi

Gin drayveridagi ([goravel/gin#222](https://github.com/goravel/gin/pull/222)) va Fiber drayveridagi ([goravel/fiber#263](https://github.com/goravel/fiber/pull/263)) timeut middleware'i bir vaqtning o'zida ishlash xatosini tuzatish uchun qayta yozildi ([goravel#966](https://github.com/goravel/goravel/issues/966)). Oldingi implementatsiya so'rov ishlovchilari zanjirini fon gorutinasida ishlatgan va so'rov sohasidagi holatni qayta ishlatish mumkin bo'lgan vaqtda timeut javobini qaytarishi mumkin edi, bu esa yuqori bir vaqtda ishlashda so'rovlararo javob ifloslanishiga sabab bo'lgan.

Ikkala drayver ham o'z frameworkining mahalliy timeut middleware'iga o'tkazadi (Gin uchun `gin-contrib/timeout`, Fiber uchun o'rnatilgan timeout), bu esa timeutga uchragan so'rov kontekstlarini qayta ishlatish o'rniga to'g'ri bekor qiladi. `Timeout(...)` middleware imzosi o'zgarishsiz va `ctx.Done()` hali ham so'rov muddatlarini aks ettiradi — hech qanday dastur kodini o'zgartirish talab qilinmaydi.
